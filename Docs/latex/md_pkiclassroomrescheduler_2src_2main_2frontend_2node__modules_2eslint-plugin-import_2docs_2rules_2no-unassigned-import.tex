With both Common\+JS\textquotesingle{} {\ttfamily require} and the ES6 modules\textquotesingle{} {\ttfamily import} syntax, it is possible to import a module but not to use its result. This can be done explicitly by not assigning the module to as variable. Doing so can mean either of the following things\+:


\begin{DoxyItemize}
\item The module is imported but not used
\item The module has side-\/effects (like \href{https://www.npmjs.com/package/should}{\texttt{ {\ttfamily should}}}). Having side-\/effects, makes it hard to know whether the module is actually used or can be removed. It can also make it harder to test or mock parts of your application.
\end{DoxyItemize}

This rule aims to remove modules with side-\/effects by reporting when a module is imported but not assigned.\hypertarget{md_pkiclassroomrescheduler_2src_2main_2frontend_2node__modules_2eslint-plugin-import_2docs_2rules_2no-unassigned-import_autotoc_md10004}{}\doxysection{\texorpdfstring{Options}{Options}}\label{md_pkiclassroomrescheduler_2src_2main_2frontend_2node__modules_2eslint-plugin-import_2docs_2rules_2no-unassigned-import_autotoc_md10004}
This rule supports the following option\+:

{\ttfamily allow}\+: An Array of globs. The files that match any of these patterns would be ignored/allowed by the linter. This can be useful for some build environments (e.\+g. css-\/loader in webpack).

Note that the globs start from the where the linter is executed (usually project root), but not from each file that includes the source.
